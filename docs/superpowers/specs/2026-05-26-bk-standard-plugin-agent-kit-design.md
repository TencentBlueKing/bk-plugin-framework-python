# BK Standard Plugin Agent Kit Design

## Overview

This design defines an agent-neutral local development kit for BlueKing standard plugin development.
The kit helps a user start from a plugin Demo project generated by the `template` cookiecutter in
`bk-plugin-framework-python`, implement a business requirement, and verify the plugin through the
same local protocol surfaces used by Standard Ops / BKFlow `remote_plugin` integration.

The first version is intentionally local-first. It does not require a shared MCP deployment, does not
create a real BK-SOPS or BKFlow task, and does not infer behavior from the latest GitHub branch.
It treats the user's generated plugin Demo project and its pinned dependencies as the source of truth.

## Goals

- Provide a reusable workflow that can be followed by Codex, Claude Code, Cursor, OpenClaw, BKAIDEV,
  other agent platforms, or a developer running commands manually.
- Guide an agent from requirement clarification to plugin implementation, unit tests, local runtime
  startup guidance, protocol-level invocation, schedule polling, and failure diagnosis.
- Keep deterministic operations in scripts so different agents call the same validation logic.
- Validate locally through `meta`, `detail`, `invoke`, and `schedule` endpoints with request bodies
  shaped like Standard Ops / BKFlow `remote_plugin` calls.
- Respect the framework and runtime versions pinned by the user's plugin Demo project.

## Non-Goals

- The first version will not deploy or operate a central MCP service.
- The first version will not create real Standard Ops / BKFlow templates, tasks, or nodes.
- The first version will not require APIGW, PaaS, BK-SOPS, or BKFlow credentials.
- The first version will not modify the framework implementation in `bk-plugin-framework-python`.
- The first version will not fetch the latest GitHub branch unless the user explicitly asks for a
  latest-version comparison.

## Project Context

The generated Demo project normally contains:

```text
app_desc.yml
bin/manage.py
bk_plugin/meta.py
bk_plugin/versions/
bk_plugin/forms/
requirements.txt
runtime.txt
tests/
```

The framework exposes the local protocol surfaces under `/bk_plugin/`:

- `GET /bk_plugin/meta/`
- `GET /bk_plugin/detail/<version>`
- `POST /bk_plugin/invoke/<version>`
- `GET /bk_plugin/schedule/<trace_id>`
- `GET /bk_plugin/logs/<trace_id>` in development mode

The Standard Ops / BKFlow `remote_plugin` integration calls `detail` to discover schema, sends
`inputs` and `context` to `invoke`, stores the returned `trace_id`, and polls `schedule` when the
plugin enters `POLL` or `CALLBACK`.

## Architecture

The kit is structured as a platform-neutral package:

```text
agent-kit/
├── AGENT.md
├── manifest.yaml
├── workflows/
│   ├── 01-orient-demo-project.md
│   ├── 02-develop-plugin-requirement.md
│   ├── 03-verify-like-bksops.md
│   └── 04-troubleshoot-failure.md
├── scripts/
│   ├── inspect_demo.py
│   ├── validate_plugin.py
│   ├── generate_version.py
│   ├── render_call_payload.py
│   ├── simulate_invoke.py
│   ├── poll_schedule.py
│   └── collect_trace.py
├── templates/
│   ├── version.py.j2
│   ├── form.js.j2
│   ├── test_plugin.py.j2
│   └── verify_payload.json.j2
└── adapters/
    ├── codex/SKILL.md
    ├── claude/CLAUDE.md
    ├── cursor/rules.md
    ├── openclaw/agent.md
    ├── bkaidev/agent.md
    └── generic/prompt.md
```

### Core Layer

`AGENT.md`, `manifest.yaml`, `workflows/`, `scripts/`, and `templates/` are the canonical
implementation of the development process. They avoid references to a specific IDE or agent vendor.

### Adapter Layer

Files under `adapters/` are thin entrypoints. They tell a platform how to load `AGENT.md`, how to read
`manifest.yaml`, and which scripts to run at each stage. They do not copy the full workflow content.

### Script Layer

Scripts are regular local commands that can be run by any agent with shell access or by a developer.
They output JSON by default and support a human-friendly `--pretty` flag.

## Workflow

### 1. Orient the Demo Project

The agent first confirms it is inside a generated plugin Demo project by checking for:

- `bk_plugin/meta.py`
- `bk_plugin/versions/`
- `bk_plugin/forms/`
- `bin/manage.py`
- `requirements.txt`

It then runs `inspect_demo.py` to collect:

- detected project root
- framework and runtime dependency pins
- locally importable framework and runtime versions, when available
- registered plugin versions
- corresponding form files
- test files
- local runtime hints

If the current directory is not a Demo project, the agent stops and asks the user for the correct
project path or suggests creating one from the cookiecutter template.

### 2. Develop the Plugin Requirement

The agent clarifies the business requirement in plugin terms:

- plugin version
- inputs visible to the plugin user
- context inputs supplied by Standard Ops / BKFlow
- outputs consumed by the flow
- synchronous, polling, or callback behavior
- external API dependencies and required secrets
- expected success and failure examples

For breaking changes, the workflow defaults to creating a new plugin version rather than changing an
existing version in place. Breaking changes include adding required inputs, removing required inputs,
changing output types, changing form data shape, or changing the meaning of the plugin behavior.

The agent may use `generate_version.py` to create a version skeleton, then fills in the business
logic and targeted tests.

### 3. Verify Like BK-SOPS Locally

The default verification path is local protocol simulation:

1. The user or agent starts the runtime, for example `python bin/manage.py rundebugserver`.
2. `simulate_invoke.py` calls `GET /bk_plugin/meta/`.
3. `simulate_invoke.py` calls `GET /bk_plugin/detail/<version>`.
4. `render_call_payload.py` builds an invoke payload using the detail schema and the verification
   fixture.
5. `simulate_invoke.py` calls `POST /bk_plugin/invoke/<version>`.
6. If the returned state is `POLL` or `CALLBACK`, `poll_schedule.py` calls
   `GET /bk_plugin/schedule/<trace_id>` until a terminal state or timeout.
7. `collect_trace.py` collects available trace logs when verification fails.

The default invoke payload includes Standard Ops / BKFlow-like context:

```json
{
  "inputs": {},
  "context": {
    "project_id": 1,
    "project_name": "local",
    "bk_biz_id": 2,
    "bk_biz_name": "local",
    "operator": "admin",
    "executor": "admin",
    "task_id": 1,
    "task_name": "local verify"
  }
}
```

The default headers include:

```text
Bkplugin-Scope-Type: space
Bkplugin-Scope-Value: local
```

The verification result is reported as local protocol simulation, not as proof that a real BK-SOPS or
BKFlow environment has executed the plugin.

### 4. Troubleshoot Failures

The agent follows a fixed failure diagnosis path:

- project structure not recognized
- framework or runtime dependency missing
- version not registered
- duplicate or invalid version
- missing `form.js`
- `Inputs` validation failed
- `ContextInputs` validation failed
- `Outputs` not written
- runtime unreachable
- `invoke` returned `FAIL`
- `schedule` row missing
- `schedule` remained unfinished until timeout
- scope permission denied
- worker or broker missing for polling and callback flows
- trace logs unavailable

Each failure path maps to script error codes and human-readable hints.

## Scripts

### `inspect_demo.py`

Detects the Demo project and returns project metadata.

Inputs:

- `--project-root`, default current directory
- `--pretty`, optional

Outputs:

- `result`
- `data.project_root`
- `data.framework_version`
- `data.runtime_version`
- `data.versions`
- `data.forms`
- `data.tests`
- `hints`

### `validate_plugin.py`

Performs static validation of plugin definitions and matching assets.

Checks:

- every plugin class inherits from the framework `Plugin`
- every plugin class has `Meta.version`
- semantic version format is valid for the current framework
- version conflicts are absent
- `Inputs`, `ContextInputs`, and `Outputs` inherit from the expected base models
- `forms/<version>/form.js` exists when expected
- generated tests exist for newly created versions when the workflow created them

### `generate_version.py`

Creates a new plugin version file, form file, and optional test file from templates.

Rules:

- never overwrite an existing file unless `--force` is explicitly passed
- default to a new version when the requested change is breaking
- preserve existing user code and unrelated files

### `render_call_payload.py`

Builds a Standard Ops / BKFlow-like invoke payload from the local detail schema and a verification
fixture.

The command supports explicit inputs and context overrides, so agents can test realistic examples
without editing script code.

### `simulate_invoke.py`

Calls local protocol endpoints using a configured base URL.

Required inputs:

- `--base-url`, for example `http://localhost:8000`
- `--version`
- `--payload-file`

Outputs include:

- `meta`
- `detail`
- `trace_id`
- `state`
- `outputs`
- `err`
- `next_action`

### `poll_schedule.py`

Polls the local schedule endpoint by `trace_id`.

It stops on `SUCCESS` or `FAIL`, or returns a timeout result with the last observed state.

### `collect_trace.py`

Collects trace-related logs when available.

It first tries the local logs endpoint in development mode. If unavailable, it checks configured or
well-known local log files. It does not require PaaS log APIs.

## Templates

Templates are minimal and explicit:

- `version.py.j2` creates a plugin class with `Meta`, `Inputs`, `ContextInputs`, `Outputs`, and a
  safe placeholder `execute` body.
- `form.js.j2` creates a version-matched form shell that maps visible inputs.
- `test_plugin.py.j2` creates tests for success, input validation, and expected failure behavior.
- `verify_payload.json.j2` creates an invoke payload fixture with Standard Ops / BKFlow-like context.

Templates should not invent complex business logic. The agent fills business logic after clarifying
the user requirement.

## Version Strategy

The kit uses this source priority:

1. The user's Demo project dependency pins, such as `requirements.txt` and `runtime.txt`.
2. Locally installed packages importable from the Demo project's Python environment.
3. A local `bk-plugin-framework-python` checkout, only when the user is explicitly developing against
   that checkout or the Demo project points to it.
4. Remote GitHub or online documentation, only when the user explicitly asks for latest-version
   comparison.

The agent must not assume latest framework behavior is available in the user's Demo project.

## Cross-Agent Compatibility

The kit has three compatibility tiers:

| Tier | Platform Capability | Supported Behavior |
| --- | --- | --- |
| Full | Read files, write files, run shell | Full develop, validate, simulate, poll, diagnose workflow |
| Partial | Read/write files, no shell | Generate and edit files, then provide commands for the user to run |
| Prompt-only | Read instructions only | Use `AGENT.md` and workflows as guidance, without automated validation |

Adapters are responsible only for translating platform conventions. The workflow and scripts remain
the stable contract.

## Error Handling

Scripts return a consistent JSON envelope:

```json
{
  "result": false,
  "code": "context_validation_failed",
  "message": "ContextInputs missing required field: executor",
  "data": {},
  "hints": [
    "Check the plugin ContextInputs definition",
    "Check whether context.executor exists in the verification payload"
  ]
}
```

Initial error codes:

- `project_not_found`
- `dependency_missing`
- `version_not_found`
- `version_conflict`
- `form_missing`
- `inputs_validation_failed`
- `context_validation_failed`
- `invoke_failed`
- `schedule_not_found`
- `schedule_failed`
- `schedule_timeout`
- `runtime_unreachable`
- `scope_denied`
- `trace_unavailable`

## Security

- Scripts must not print app secrets, gateway tokens, cookies, private tokens, or authorization
  headers.
- Local verification defaults to non-sensitive fake context values.
- The first version does not call real APIGW, PaaS, BK-SOPS, or BKFlow APIs.
- Reports must mark their scope as local protocol simulation.

## Testing

The implementation plan should include:

- unit tests for script argument parsing and JSON envelopes
- unit tests for Demo project detection and version parsing
- unit tests for payload rendering
- fixture tests using a minimal cookiecutter-style Demo project
- local HTTP tests against a running debug server where feasible
- failure-mode tests for missing version, invalid payload, unreachable runtime, and schedule timeout

## Deliverables

The first implementation should produce:

- `AGENT.md`
- `manifest.yaml`
- workflow documents
- scripts listed in this design
- templates listed in this design
- thin adapters for at least `generic` and `codex`
- tests for script behavior
- a sample verification report format

After a successful agent-assisted plugin development session, the expected project-level outputs are:

- a version file such as `bk_plugin/versions/v1_1_0.py`
- a matching form file such as `bk_plugin/forms/1.1.0/form.js`
- focused tests such as `tests/test_plugin_v1_1_0.py`
- a verification report containing version, trace id, final state, outputs, error message, payload
  path, and verification time

## Open Decisions Resolved in This Design

- The first version uses an agent-neutral local kit rather than a shared MCP service.
- The first version targets generated plugin Demo projects rather than the framework source tree.
- Verification uses local protocol simulation rather than a real BK-SOPS or BKFlow environment.
- Most agent integrations are thin adapters over shared workflows and scripts.
